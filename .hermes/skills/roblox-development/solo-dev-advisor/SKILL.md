# Solo Developer Vision

## Why I Am Doing This

My goal is not to build a single successful Roblox game.

My goal is to build a sustainable game development platform that allows me to create, operate, and grow multiple Roblox games as a solo developer.

Every decision should be evaluated not only by how much it helps the current game, but by how much it helps future games.

I am not building Scraplands.

I am building the systems, tools, workflows, and knowledge that make Scraplands possible—and that will make future games faster to create.

---

# Long-Term Goals

## Goal 1: Build Multiple Games

Success is not measured by how large Scraplands becomes.

Success is measured by whether I can launch additional games without starting over.

Every new game should become easier to create than the previous one.

---

## Goal 2: Build Reusable Systems

Whenever possible, game-specific code should evolve into reusable packages.

Examples include:

- Save systems
- Economy systems
- Event systems
- Trading systems
- Notification systems
- Daily rewards
- Analytics
- Localization
- AI workflows
- Live operations tools

The ideal future is that a new game can be started with a large percentage of the foundation already complete.

---

## Goal 3: Reduce Context Switching

I should spend as little mental energy as possible remembering:

- What I was working on
- What needs to happen next
- What bugs exist
- What features are planned

Project state should live in Hermes.

My brain should focus on decisions, creativity, and problem solving.

---

## Goal 4: Prioritize Live Operations

Large permanent systems increase complexity.

Events create engagement without permanently increasing maintenance costs.

Whenever possible:

- Build reusable event frameworks
- Reuse systems across events
- Deliver content through events rather than new permanent mechanics

---

## Goal 5: Measure Impact

Features are only successful if they improve player outcomes.

Prioritize:

- Retention
- Session length
- Monetization
- Engagement
- Player satisfaction

Avoid spending significant time polishing features that do not meaningfully affect player behavior.

---

## Goal 6: Ship More Frequently

Perfect is not the goal.

Progress is the goal.

A good feature released today is usually more valuable than a perfect feature released weeks later.

Before spending significant time polishing something, ask:

"Will most players notice the difference?"

If the answer is no, ship it.

---

## Goal 7: Protect Creative Energy

My highest value work is:

- Game design
- Progression design
- Economy design
- Monetization design
- Event design
- Product strategy

My lowest value work is:

- Manual triage
- Administrative work
- Repetitive documentation
- Routine maintenance
- Mechanical coding tasks

Whenever possible, repetitive work should be delegated to automation, AI, or reusable systems.

---

# Development Principles

## Think Platform First

Before implementing a system, ask:

"Could this be reused in another game?"

If yes, design it accordingly.

---

## Build Once, Use Many Times

Favor reusable frameworks over one-off solutions.

Short-term convenience should not outweigh long-term leverage.

---

## Simplicity Wins

Prefer:

- Smaller systems
- Fewer dependencies
- Easier maintenance
- Clear ownership

Complexity is a cost that compounds forever.

---

## Data Over Assumptions

Use player behavior to guide decisions.

Do not assume a feature is valuable simply because it took time to build.

---

## Done Beats Perfect

Do not block releases on:

- Minor UI polish
- Naming debates
- Small balance improvements
- Tiny copy refinements

Ship, learn, improve.

---

# Definition of Success

A successful month is not measured by:

- Lines of code written
- Features added
- Hours worked

A successful month is measured by:

- Systems reused
- Time saved
- Features shipped
- Players retained
- Revenue generated
- Progress toward future games

---

# The Question To Ask Before Any Work

Before starting a task, ask:

"Is this helping me build a game, or helping me build a game studio?"

Prefer the option that moves both forward.
